## 1. Что за проект и статус

**MultiAgency DeepResearch** — автономный агент-исследователь на **OpenAI Agents
SDK** с UI на **Gradio**. Пользователь задаёт тему → агент уточняет детали →
автономно исследует → пишет отчёт с самопроверкой качества → показывает результат.
Развитие учебного примера (последовательный вызов агентов) в настоящую агентную
систему с автономией и контролем качества.

- Название (D-11): display — **MultiAgency DeepResearch**; slug —
  **`multiagency-deep-research`** (git-репо, папка, HF Space).
- Под контролем **git**, ветка **main**. **HEAD `6bdaad5` на `origin/main`,
  дерево чистое.**
- Стек: Python, `agents` (OpenAI Agents SDK), Pydantic (структурированный вывод),
  Gradio (версия зафиксирована в `requirements.txt`), `asyncio` (параллельные
  поиски), встроенный трейсинг SDK.
- **Gate 0-3 — DONE. Gate 4 (Evaluator-Optimizer петля) — следующий, не открыт.**

## 3. Runtime-архитектура (три зоны)

- **Зона 1 — Прелюдия (человек в цикле, ВНЕ автономии).** `clarifier_agent`
  задаёт ровно 3 вопроса → UI ждёт ответы → `build_brief(topic, Q&A) -> str`
  (D-10: бриф — плоская строка). Бриф — единый источник намерения. **← Gate 1, DONE.**
- **Зона 2 — Автономное ядро (решает LLM).** `manager_agent` вызывает
  специалистов как инструменты (`.as_tool()` / `@function_tool`): планировщик,
  параллельный поиск, писатель. Сам решает число и направление поисков, включая
  динамическую вторую волну по пробелам. Паттерн — **agents-as-tools** (D-01),
  менеджер владеет финальным ответом (эмитит `ReportData` через `output_type` —
  D-12/B1). НЕ handoffs. **← Gate 3, DONE.**
- **Зона 3 — Контроль качества (детерминированный код).** `evaluator_agent`
  оценивает отчёт; петля дорабатывает в `while` до `score >= 7` либо `MAX_ROUNDS = 2`.
  Петля — в Python (D-02), НЕ у менеджера. **← это Gate 4, ещё не создан.**

## 4. Принятые решения (кратко; полностью — DECISIONS.md)

- **D-01** — agents-as-tools как основной паттерн (менеджер владеет процессом).
  *Уточнение (D-12):* B1 реализует «менеджер владеет финальным ответом» буквально —
  финал эмитит менеджер через `output_type=ReportData`.
- **D-02** — evaluator-петля живёт в коде, не у менеджера (гарантия лимита).
- **D-03** — предохранители обязательны (`max_turns`, бюджет поисков, порог+лимит).
- **D-06** — E2E-контракт: один happy-path на моках, проверяет сборку, не качество.
- **D-07** — контекст: 150K предупреждение / 170K потолок.
- **D-09** — один изолированный handoff на `finalizer_agent` на финальном
  оформлении, ПОСЛЕ evaluator-петли. *Уточнение (граница на 3d):* handoff
  откладывается на стык с Gate 4 — реализуется вместе с петлёй; при B1 финализатор
  получает готовый `ReportData`.
- **D-10** — бриф передаётся как `str`.
- **D-11** — название/slug (см. §1).
- **D-12** — контракт финала менеджера: B1 (`output_type=ReportData`); менеджер
  сам эмитит структуру, SDK валидирует. B2 (`StopAtTools(["write_report"])`)
  **отклонён**: `.as_tool()` стрингифицирует вывод писателя (`text_message_outputs`),
  `StopAtTools` вернул бы `str`, а не `ReportData`.

## 8. Вход Gate 4 — критические пункты (НЕ потерять)

1. **Создать `evaluator_agent`** с `output_type` (Pydantic): `score`, `passed`,
   `feedback`, `missing_topics`.
2. **Петля — в КОДЕ, не у менеджера (D-02):** детерминированный
   `while not passed and rounds < MAX_ROUNDS` (`MAX_ROUNDS = 2`), порог
   `score >= 7` (не 10), graceful exit — при исчерпании лимита наружу идёт лучшая
   версия по score. Петля даёт гарантию лимита; менеджер может «забыть» счётчик.
3. **Каждая итерация отдаёт наружу `score` и `feedback`** (задел под UI, Gate 5).
4. **Расширить E2E:** труба проходит через оценщик, остаётся зелёной.
5. **D-09 / finalizer — принести владельцу явно.** На Gate 4 появляется
   evaluator-петля, ПОСЛЕ утверждения отчёта которой по D-09/D-12 ложится
   отложенный **handoff на `finalizer_agent`**. Финализатор получает готовый
   `ReportData` (следствие B1). Где именно ложится граница handoff — **решение
   владельца (RULES §1)**, приносится при постройке петли; Claude Code НЕ решает
   сам. Один изолированный handoff, только финальное оформление, не участвует в
   оценке.

## 9. Критерии Gate 4 (дословно из WORKFLOW.md)

- [ ] `evaluator_agent` с `output_type`: `score`, `passed`, `feedback`, `missing_topics`.
- [ ] Петля: `while not passed and rounds < MAX_ROUNDS` (`MAX_ROUNDS = 2`).
- [ ] Порог `score >= 7` (не 10).
- [ ] Graceful exit: лимит исчерпан → лучшая версия по score наружу.
- [ ] Каждая итерация отдаёт наружу `score` и `feedback` (для UI).
- [ ] Юнит-тесты: выход по порогу; выход по лимиту; выбор лучшей версии.
- [ ] E2E расширен: труба проходит через оценщик.

**Тесты:** unit (три сценария петли) + E2E (труба с оценщиком).
**Не тестируем:** «умность» самих оценок LLM — недетерминированны.

## 14. Техзаметки

- Запуск тестов: `python -m pytest -v` (конфиг `pytest.ini`: `pythonpath = src`,
  `testpaths = tests`). Импорт агентов не требует ключа OpenAI (реальных вызовов нет).
- `RunContextWrapper[T]` как первый параметр `@function_tool` инжектится SDK и
  **скрыт от LLM**.
- Мок-слой патчит `agents.Runner.run` на классе. Менеджер прогоняется настоящим
  `Runner` со сценарной моделью (`RunConfig(model=ScenarioManagerModel(...))`);
  `.as_tool()`-извлечение идёт через `result.new_items` + `text_message_outputs`,
  поэтому двойник RunResult несёт `.new_items` с настоящим `MessageOutputItem`.
- `final_output_as(cls)` без `raise_if_incorrect_type` — чистый `cast`; чтобы
  `isinstance(report, ReportData)` держался, менеджер эмитит финал через
  `output_type=ReportData` (D-12).
- Локально: `.env` с `OPENAI_API_KEY`, затем `python app.py` (сейчас `app.py` —
  заглушка-точка входа). `.env` в `.gitignore`, в репо только `.env.example`.
