## 1. Что за проект и статус

**MultiAgency DeepResearch** — автономный агент-исследователь на **OpenAI Agents
SDK** с UI на **Gradio**. Пользователь задаёт тему → агент уточняет детали →
автономно исследует → пишет отчёт с самопроверкой качества → показывает результат.
Развитие учебного примера (последовательный вызов агентов) в настоящую агентную
систему с автономией и контролем качества.

- Название (D-11): display — **MultiAgency DeepResearch**; slug —
  **`multiagency-deep-research`** (git-репо, папка, HF Space).
- Под контролем **git**, ветка **main**. Дерево чистое.
- Стек: Python, `agents` (OpenAI Agents SDK), Pydantic (структурированный вывод),
  Gradio (версия зафиксирована в `requirements.txt`), `asyncio` (параллельные
  поиски), встроенный трейсинг SDK.

## 3. Runtime-архитектура (три зоны)

- **Зона 1 — Прелюдия (человек в цикле, ВНЕ автономии).** `clarifier_agent`
  задаёт ровно 3 вопроса → UI ждёт ответы → `build_brief(topic, Q&A) -> str`
  (D-10: бриф — плоская строка). Бриф — единый источник намерения.
- **Зона 2 — Автономное ядро (решает LLM).** `manager_agent` вызывает
  специалистов как инструменты (`.as_tool()` / `@function_tool`): планировщик,
  параллельный поиск, писатель. Сам решает число и направление поисков, включая
  динамическую вторую волну по пробелам. Паттерн — **agents-as-tools** (D-01),
  менеджер владеет финальным ответом. НЕ handoffs. **← это Gate 3, ещё не создан.**
- **Зона 3 — Контроль качества (детерминированный код).** `evaluator_agent`
  оценивает отчёт; петля дорабатывает в `while` до `score >= 7` либо `MAX_ROUNDS = 2`.
  Петля — в Python (D-02), НЕ у менеджера. **← это Gate 4.**

## 4. Принятые решения (кратко; полностью — DECISIONS.md)

- **D-01** — agents-as-tools как основной паттерн (менеджер владеет процессом).
- **D-02** — evaluator-петля живёт в коде, не у менеджера (гарантия лимита).
- **D-03** — предохранители обязательны (`max_turns`, бюджет поисков, порог+лимит).
- **D-06** — E2E-контракт: один happy-path на моках, проверяет сборку, не качество.
- **D-07** — контекст: 150K предупреждение / 170K потолок.
- **D-09** — один изолированный handoff на `finalizer_agent` на финальном
  оформлении, ПОСЛЕ evaluator-петли. НЕ участвует в поиске/оценке.
- **D-10** — бриф передаётся как `str`.
- **D-11** — название/slug (см. §1).

## 7. Что готово (итог Gate 2)

**Три агента-инструмента (Зона 2, agents-as-tools, D-01):**
- `plan_web_searches` — `planner_agent.as_tool()` в [src/planner_agent.py](src/planner_agent.py).
  Схема `WebSearchPlan` / `WebSearchItem`.
- `run_parallel_searches` — `@function_tool` в [src/search_agent.py](src/search_agent.py):
  гоняет `Runner.run(search_agent, ...)` по плану параллельно. Устойчив — падение
  одного поиска → пропуск (None), пачка не рушится; `asyncio.as_completed`.
  Выбор `@function_tool` (не `.as_tool()`), т.к. это оркестрация НАД агентом.
- `write_report` — `writer_agent.as_tool()` в [src/writer_agent.py](src/writer_agent.py).
  Схема `ReportData` (`short_summary`, `markdown_report`, `follow_up_questions`).

**Предохранители — [src/guardrails.py](src/guardrails.py) (D-03):**
- `ResearchContext` — бюджет поисков; **списание по попыткам, не по успехам**
  (упавший поиск тоже тратит бюджет); при исчерпании — **контролируемый отказ**
  (сигнал `SEARCH_BUDGET_EXHAUSTED`, а НЕ исключение); пачка больше остатка —
  капается (`SEARCH_BUDGET_PARTIAL`). Считается по отдельным поискам.
- Константы `DEFAULT_SEARCH_BUDGET = 12`, `MAX_TURNS = 15` — **приняты владельцем**.
- Бюджет менеджеру передаётся через `RunContextWrapper[ResearchContext]` (скрыт
  от LLM). **Пока к рантайму НЕ подключён** — это вход Gate 3 (см. §8).

**Каркас E2E (D-06):**
- Мок-слой: [tests/_mocks.py](tests/_mocks.py) (`make_plan`/`make_report`,
  `search_run`, `MockAgentRunner` — диспетчер `Runner.run` по имени агента,
  проброс неизвестного агента к настоящему Runner) + [tests/conftest.py](tests/conftest.py)
  (фикстура `mock_agent_runner`).
- Скелет: [tests/test_e2e_pipeline.py](tests/test_e2e_pipeline.py) — happy-path
  `бриф→план→поиски→отчёт`, написан под настоящий рантайм, `xfail(strict=True)`
  до появления менеджера.

**Тесты: 38 passed, 1 xfailed** (`python -m pytest -v`, на моках, без ключа).

## 8. Вход Gate 3 — критические пункты (НЕ потерять)

1. **Создать `manager_agent`** (agents-as-tools, D-01) с инструментами: план,
   параллельный поиск, писатель. **Промпт менеджера явно задаёт**: бюджет поисков,
   реакцию на `SEARCH_BUDGET_EXHAUSTED`, инструкцию про динамическую вторую волну
   по пробелам в данных (CLAUDE.md).
2. **Подключить предохранители к рантайму:**
   `Runner.run(manager_agent, brief, context=ResearchContext(), max_turns=MAX_TURNS)`.
   **Без этого 2d — мёртвый код** (бюджет не читается, потолок не действует).
   E2E-скелет это проверяет.
3. **Оживить E2E:** создать менеджера → **снять `xfail`**. `strict=True` → в момент
   сборки трубы xpass упадёт — это осознанный сигнал снять маркер. **Скелет НЕ
   переписывать, только оживить.**
4. **DEFERRED #4:** возможна доработка двойника RunResult в мок-слое при извлечении
   вывода `.as_tool()`-обёрток самим SDK при вызове менеджером. Решить при
   оживлении E2E.
5. **D-09 (handoff на финал) — уточнить границу, НЕ решать самому.** Handoff на
   `finalizer_agent` относится к Gate 3, но НЕ участвует в поиске/оценке: один
   изолированный handoff только на финальном оформлении, ПОСЛЕ evaluator-петли
   (петля — Gate 4). На Gate 3 закладывается автономное ядро; полноценный
   финализатор может лечь на стык с Gate 4. **Принести на обсуждение владельцу при
   открытии Gate 3** (RULES §1).

## 9. Критерии Gate 3 (дословно из WORKFLOW.md)

- [ ] Промпт менеджера: бюджет поисков, инструкция про вторую волну по пробелам.
