# MultiAgency DeepResearch

Автономный агент-исследователь на **OpenAI Agents SDK** с интерфейсом на **Gradio**.
Пользователь задаёт тему → агент уточняет детали → автономно исследует веб →
пишет отчёт с самопроверкой качества → показывает результат.

Проект развивает учебный базовый пример (последовательный вызов агентов) в
сторону настоящей агентной системы с автономией, контролем качества и
человеком в цикле. Технический slug — `multiagency-deep-research` (D-11).

## Возможности

- Уточняющие вопросы перед исследованием (человек в цикле).
- Автономный менеджер, сам решающий число и направление поисков.
- Параллельный веб-поиск.
- Контроль качества отчёта по паттерну Evaluator–Optimizer.
- Пошаговый интерфейс с живым прогрессом.

## Архитектура (три зоны)

**Прелюдия (человек в цикле).** `clarifier_agent` задаёт 3 вопроса; ответы
склеиваются с темой в единый бриф — источник намерения для всего пайплайна.

**Автономное ядро.** `manager_agent` вызывает специалистов как инструменты
(agents-as-tools): планировщик, параллельный поиск, писатель. Сам решает, сколько
и каких поисков провести, включая динамическую вторую волну по пробелам в данных.

**Контроль качества.** `evaluator_agent` оценивает отчёт; детерминированная петля
в коде дорабатывает его до порога качества либо до лимита итераций.

### О паттернах: agents-as-tools и handoffs

Основной паттерн оркестрации — **agents-as-tools** (D-01): менеджер остаётся
владельцем процесса и финального результата, вызывая агентов как инструменты. Это
осознанный выбор — согласно документации OpenAI, конвейеры обработки (а это именно
наш случай: тема → план → поиск → отчёт → оценка) ложатся на agents-as-tools лучше,
чем на handoffs, и проще отлаживаются. Handoffs в середине конвейера усложнили бы
отладку без пользы.

**Handoff** используется точечно в одном месте (D-09) — на финальном оформлении
отчёта, после утверждения его оценщиком (`finalizer_agent`). Это демонстрирует
владение обоими паттернами, не нарушая контроль менеджера над основной работой:
после финального оформления в пайплайне ничего не происходит, поэтому единственная
передача безопасна.

## Стек

Python 3.10+, OpenAI Agents SDK (`openai-agents`), Pydantic, Gradio, `asyncio`.

## Запуск локально

```bash
python -m venv .venv
# Windows: .venv\Scripts\activate | macOS/Linux: source .venv/bin/activate
pip install -r requirements.txt

cp .env.example .env   # затем впиши OPENAI_API_KEY
python app.py
```

## Деплой (Hugging Face Spaces)

Приложение разворачивается как **Docker-Space** (`sdk: docker`, `app_port: 7860`;
D-16). Docker — сознательный выбор: при `sdk: gradio` HF впрыскивает
`gradio[oauth,mcp]`, чей extra `mcp` пинит `mcp==1.10.1` и конфликтует с
`openai-agents==0.4.2` (требует `mcp>=1.11.0`). MCP/OAuth проект не использует, поэтому
`Dockerfile` ставит **голый** `gradio==5.49.1` из `requirements.txt` — `mcp` тянет
только `openai-agents`. Оба жёстких пина сохранены, версии не менялись. `Dockerfile`
задаёт `GRADIO_SERVER_NAME=0.0.0.0` и `GRADIO_SERVER_PORT=7860` через окружение —
`app.py` не меняется, точка входа прежняя.

**Секреты — только через настройки Space, не в коде (RULES §5).**
`OPENAI_API_KEY` заводится как **Secret** в настройках Space (Settings → Variables
and secrets), а НЕ хранится в репозитории и НЕ хардкодится. SDK читает ключ из
окружения сам. Файл `.env` — в `.gitignore`; в репозитории только `.env.example`.

Отправка email на Space не выполняется — чекбокс копии остаётся демонстрацией
(D-05); адреса не хардкодятся, отдельный секрет не заводится.

## Тесты

```bash
pip install -r requirements-dev.txt   # pytest и пр. — на Spaces не ставятся
python -m pytest
```

Тесты используют моки LLM — без реальных вызовов OpenAI и без ключа. Deploy- и
dev-зависимости разнесены: `requirements.txt` (рантайм Space, жёсткие пины) и
`requirements-dev.txt` (тесты и legacy-демо).

## Структура

```
app.py                точка входа (Gradio)
src/                  агенты и оркестрация
tests/                тесты на моках
docs/                 документы процесса разработки (WORKFLOW, DECISIONS, RULES, …)
handover/             handover-документы по gates
requirements.txt      deploy-зависимости (рантайм Space)
requirements-dev.txt  dev/test-зависимости (не на Spaces)
```

## Документация процесса

Разработка ведётся по gates с контролем качества. Ключевые документы:
`docs/WORKFLOW.md` (карта gates), `docs/DECISIONS.md` (архитектурные решения),
`docs/RULES.md` (инварианты проекта), `CLAUDE.md` (контекст для Claude).
