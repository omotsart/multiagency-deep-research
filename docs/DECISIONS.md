### D-01 — Agents-as-tools как основной паттерн
- Статус: ПРИНЯТО
- Контекст: ТЗ просит «внедрить handoffs», но проект — пайплайн обработки, не разговор с маршрутизацией.
- Решение: основной паттерн — agents-as-tools; менеджер владеет процессом и финальным ответом.
- Причина: официальная документация OpenAI прямо относит пайплайны к agents-as-tools; handoffs в середине конвейера усложняют отладку без пользы.
- Последствия: ядро (план/поиск/писатель/оценщик) строится на agents-as-tools. Точечный handoff — см. D-09.

### D-09 — Один изолированный handoff на финальном оформлении
- Статус: ПРИНЯТО (закрывает Q-01, вариант «б»)
- Контекст: ТЗ буквально требует handoff; нужно выполнить пункт, не ломая архитектуру.
- Решение: после утверждения отчёта evaluator-петлёй менеджер делает handoff на `finalizer_agent`, оформляющий финальную выдачу. Один агент, одна передача, одна точка в коде.
- Причина: демонстрирует владение механикой handoff; финальная точка безопасна — после неё в пайплайне ничего не происходит, контроль менеджера над основной работой не нарушается.
- Границы: handoff НЕ участвует в поиске и оценке; только финальное оформление. Затрагивает Gate 3, не Gate 2.
- Последствия: в README объясняем оба паттерна (почему основа — agents-as-tools, зачем один handoff).

### D-02 — Evaluator-Optimizer петля живёт в коде, не у менеджера
- Статус: ПРИНЯТО
- Решение: петля качества — детерминированный `while` в Python, а не инструмент автономного менеджера.
- Причина: гарантированный лимит итераций; менеджер может «забыть» счётчик.
- Последствия: автономию демонстрируем на уровне поисков; критическую петлю держим под кодовым контролем.

### D-03 — Предохранители автономности обязательны
- Статус: ПРИНЯТО
- Решение: `max_turns`, бюджет поисков в `RunContextWrapper`, порог+лимит в evaluator, graceful exit.
- Причина: автономный агент без лимитов — неконтролируемый расход токенов/времени.

### D-04 — UI: Подход А (3 статичных текстбокса), без @gr.render
- Статус: ПРИНЯТО
- Решение: заранее созданные 3 текстбокса, включаются через `visible`; переменное число полей не делаем.
- Причина: `@gr.render` даёт риск «перекошенного» UI и сложную привязку событий; для 3 фиксированных вопросов избыточен.

### D-05 — Email в деплое: галочка без реальной отправки
- Статус: ПРИНЯТО
- Решение: чекбокс остаётся в UI как демонстрация; фактическая отправка не реализуется на Spaces.
- Причина: SendGrid на публичном Space ненадёжен; не нужен лишний секрет; захардкоженные адреса удаляем.

### D-06 — E2E-контракт: happy-path на моках
- Статус: ПРИНЯТО
- Решение: один сквозной интеграционный тест на замоканных LLM-вызовах; проверяет сборку трубы, не качество текста.
- Причина: реальные вызовы дороги, недетерминированны, требуют ключа в CI.

### D-07 — Порог контекста сессии: 150K предупреждение / 170K потолок
- Статус: ПРИНЯТО
- Решение: при ~150K и незакрытом gate — Claude пишет handover; 170K — жёсткий потолок.
- Причина: буфер на сам handover до достижения потолка.

### D-08 — Набор документов процесса
- Статус: ПРИНЯТО
- Решение: WORKFLOW, CLAUDE, DECISIONS, RULES, HANDOVER + роли gate-reviewer, test-writer. Не заводим AGENTS, SKILL, normalizer-reviewer.
- Причина: баланс контроля и простоты для небольшого проекта.

### D-10 — Бриф передаётся как плоская строка (str)
- Статус: ПРИНЯТО
- Дата: 2026-07-01
- Контекст: формат брифа — контракт данных между Зоной 1 и менеджером (Gate 3).
- Решение: `build_brief` возвращает `str`.
- Причина: единственный потребитель — менеджер, читает бриф целиком как текст намерения, выборочный доступ к полям не требуется; Pydantic-структура была бы слоем без необходимости (RULES §6, дух D-06).
- Последствия: если на Gate 3 менеджеру понадобятся отдельные поля брифа — решение заменяется (D-10 → ЗАМЕНЕНО) под зафиксированную потребность.

### D-11 — Название проекта: MultiAgency DeepResearch
- Статус: ПРИНЯТО
- Дата: 2026-07-01
- Контекст: базовое имя «Deep Research» не отражало мультиагентную суть системы.
- Решение: отображаемое имя — MultiAgency DeepResearch (README, заголовок UI); технический slug — multiagency-deep-research (git-репо, папка проекта, HF Space, kebab-case для совместимости с хостингами).
- Границы: переименование касается ТОЛЬКО обёртки (репо/Space/README/заголовок UI). Внутренние имена агентов, модулей и файлов src/ НЕ меняются — они отражают роль, не бренд.
- Последствия: git init на Gate 2 использует slug; README и заголовок UI обновляются на своих gates (5/6).
