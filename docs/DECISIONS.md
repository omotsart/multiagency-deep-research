### D-01 — Agents-as-tools как основной паттерн
- Статус: ПРИНЯТО
- Контекст: ТЗ просит «внедрить handoffs», но проект — пайплайн обработки, не разговор с маршрутизацией.
- Решение: основной паттерн — agents-as-tools; менеджер владеет процессом и финальным ответом.
- Причина: официальная документация OpenAI прямо относит пайплайны к agents-as-tools; handoffs в середине конвейера усложняют отладку без пользы.
- Последствия: ядро (план/поиск/писатель/оценщик) строится на agents-as-tools. Точечный handoff — см. D-09.
- Уточнение (D-12, 3c): B1 реализует «менеджер владеет финальным ответом» буквально — финал эмитит менеджер через `output_type=ReportData`.

### D-09 — Один изолированный handoff на финальном оформлении
- Статус: ПРИНЯТО (закрывает Q-01, вариант «б»)
- Контекст: ТЗ буквально требует handoff; нужно выполнить пункт, не ломая архитектуру.
- Решение: после утверждения отчёта evaluator-петлёй менеджер делает handoff на `finalizer_agent`, оформляющий финальную выдачу. Один агент, одна передача, одна точка в коде.
- Причина: демонстрирует владение механикой handoff; финальная точка безопасна — после неё в пайплайне ничего не происходит, контроль менеджера над основной работой не нарушается.
- Границы: handoff НЕ участвует в поиске и оценке; только финальное оформление. Затрагивает Gate 3, не Gate 2.
- Последствия: в README объясняем оба паттерна (почему основа — agents-as-tools, зачем один handoff).
- Уточнение (граница определена на 3d): handoff на `finalizer_agent` откладывается на стык с Gate 4 — реализуется вместе с evaluator-петлёй (D-02), после утверждения которой он по D-09 и идёт; в Gate 3 не строится (иначе нарушен порядок D-09 либо код без контракта, RULES §6). При B1 (D-12) терминал менеджера — готовый `ReportData`, финализатор получит структуру.
- Уточнение (механика стыка, 4d — вариант A, ПРИНЯТО владельцем): настоящий SDK-handoff инициируется ходом агента ВНУТРИ `Runner.run` (`agents/handoffs.py`), а evaluator-петля (D-02) — код, уже вернувший `ReportData`. Поэтому «handoff строго ПОСЛЕ петли» реализуется через тонкий агент-инициатор `finalizer_gateway` (`handoffs=[handoff(finalizer_agent)]`): после выхода из петли `research_and_finalize()` делает единственный прогон шлюза, тот делает transfer на `finalizer_agent`; готовый `ReportData` доходит через историю диалога. Одна точка, ВНЕ цикла; петля про финализатор не знает (`evaluator_loop.py` не тронут), лимит её итераций не затронут. Отклонены: B (handoff у менеджера — сработал бы внутри цикла, ломает «после петли»/D-02/B1) и C (прямой прогон финализатора — не задействует механику handoff, требуемую D-09). Цена A: +1 агент-шлюз сверх `finalizer_agent` и +1 недетерминированный ход LLM (только запускает transfer) — принята осознанно ради буквального выполнения D-09.
- Уточнение (природа финализатора, 4d — ПРИНЯТО владельцем): `finalizer_agent` — настоящий LLM-агент. Он принимает управление через настоящий handoff и САМ производит финальное оформление; получательство handoff = ход модели, поэтому содержание (`markdown_report`/`short_summary`/`follow_up_questions`) идёт ЧЕРЕЗ модель (финализатор восстанавливает поля из пришедшего через историю диалога `model_dump_json()` — парсинга в коде на принимающей стороне нет). Ранее обсуждавшаяся детерминированная граница «косметика в коде» (раскладка полей чистой функцией без LLM) — ОТКЛОНЕНА: несовместима с механикой handoff. Рассмотрены и отклонены две развязки, сохраняющие детерминированную раскладку: (1) «церемониальный финализатор» — нести `ReportData` в `RunContextWrapper` и раскладывать в `on_handoff`-колбэке кодом; отклонена, т.к. агент-получатель всё равно исполняет свой ход LLM, а его вывод пришлось бы игнорировать — финализатор вырождается; (3) отказ от handoff (прямой прогон финализатора) — отклонена как не-handoff (это вариант C, против буквы D-09). Следствие для тестов: целостность содержания через сам transfer юнит-тестом непокрываема (недетерминированный вывод LLM, RULES §4) — проверяется РУЧНОЙ приёмкой на живом прогоне (Gate 5/6). Мёртвый код детерминированной раскладки (`build_final_presentation` и пять косметических тестов) удалён; юнит-тестами держатся только механика и порядок стыка (handoff настоящий; строго после петли; лимит цел).

### D-02 — Evaluator-Optimizer петля живёт в коде, не у менеджера
- Статус: ПРИНЯТО
- Решение: петля качества — детерминированный `while` в Python, а не инструмент автономного менеджера.
- Причина: гарантированный лимит итераций; менеджер может «забыть» счётчик.
- Последствия: автономию демонстрируем на уровне поисков; критическую петлю держим под кодовым контролем.

### D-03 — Предохранители автономности обязательны
- Статус: ПРИНЯТО
- Решение: `max_turns`, бюджет поисков в `RunContextWrapper`, порог+лимит в evaluator, graceful exit.
- Причина: автономный агент без лимитов — неконтролируемый расход токенов/времени.

### D-04 — UI: Подход А (3 статичных текстбокса), без @gr.render
- Статус: ПРИНЯТО
- Решение: заранее созданные 3 текстбокса, включаются через `visible`; переменное число полей не делаем.
- Причина: `@gr.render` даёт риск «перекошенного» UI и сложную привязку событий; для 3 фиксированных вопросов избыточен.

### D-05 — Email в деплое: галочка без реальной отправки
- Статус: ПРИНЯТО
- Решение: чекбокс остаётся в UI как демонстрация; фактическая отправка не реализуется на Spaces.
- Причина: SendGrid на публичном Space ненадёжен; не нужен лишний секрет; захардкоженные адреса удаляем.

### D-06 — E2E-контракт: happy-path на моках
- Статус: ПРИНЯТО
- Решение: один сквозной интеграционный тест на замоканных LLM-вызовах; проверяет сборку трубы, не качество текста.
- Причина: реальные вызовы дороги, недетерминированны, требуют ключа в CI.

### D-07 — Порог контекста сессии: 150K предупреждение / 170K потолок
- Статус: ПРИНЯТО
- Решение: при ~150K и незакрытом gate — Claude пишет handover; 170K — жёсткий потолок.
- Причина: буфер на сам handover до достижения потолка.

### D-08 — Набор документов процесса
- Статус: ПРИНЯТО
- Решение: WORKFLOW, CLAUDE, DECISIONS, RULES, HANDOVER + роли gate-reviewer, test-writer. Не заводим AGENTS, SKILL, normalizer-reviewer.
- Причина: баланс контроля и простоты для небольшого проекта.

### D-10 — Бриф передаётся как плоская строка (str)
- Статус: ПРИНЯТО
- Дата: 2026-07-01
- Контекст: формат брифа — контракт данных между Зоной 1 и менеджером (Gate 3).
- Решение: `build_brief` возвращает `str`.
- Причина: единственный потребитель — менеджер, читает бриф целиком как текст намерения, выборочный доступ к полям не требуется; Pydantic-структура была бы слоем без необходимости (RULES §6, дух D-06).
- Последствия: если на Gate 3 менеджеру понадобятся отдельные поля брифа — решение заменяется (D-10 → ЗАМЕНЕНО) под зафиксированную потребность.

### D-11 — Название проекта: MultiAgency DeepResearch
- Статус: ПРИНЯТО
- Дата: 2026-07-01
- Контекст: базовое имя «Deep Research» не отражало мультиагентную суть системы.
- Решение: отображаемое имя — MultiAgency DeepResearch (README, заголовок UI); технический slug — multiagency-deep-research (git-репо, папка проекта, HF Space, kebab-case для совместимости с хостингами).
- Границы: переименование касается ТОЛЬКО обёртки (репо/Space/README/заголовок UI). Внутренние имена агентов, модулей и файлов src/ НЕ меняются — они отражают роль, не бренд.
- Последствия: git init на Gate 2 использует slug; README и заголовок UI обновляются на своих gates (5/6).

### D-12 — Контракт финала менеджера: B1 (output_type=ReportData)
- Статус: ПРИНЯТО
- Дата: 2026-07-02
- Контекст: на 3c для E2E-контракта `isinstance(report, ReportData)` понадобилось определить, что именно возвращает менеджер как результат прогона.
- Решение: `output_type=ReportData` на менеджере (вариант B1) — менеджер сам эмитит финальную структуру, SDK валидирует его финальное сообщение в `ReportData`.
- Причина: `.as_tool()` стрингифицирует вывод суб-агента (`ItemHelpers.text_message_outputs`), поэтому B2 (`tool_use_behavior=StopAtTools(["write_report"])`) вернул бы `str`, а не `ReportData` — механически не выполняет контракт. B1 проще (RULES §6) и буквально реализует D-01. Вариант B2 рассмотрен и отклонён.
- Последствия: терминал прогона — `ReportData` от менеджера. Влияет на D-09: финализатор получит готовую структуру, а не сырой выход.

### D-13 — Контракт слоя петли под UI (per-loop бюджет / канал-генератор / failed-терминал)
- Статус: ПРИНЯТО
- Дата: 2026-07-03
- Владелец подтвердил.
- Контекст: Gate 5 (UI) открывается под-гейтом 5.0 — контракт слоя петли ложится
  ДО вёрстки экранов. Пакет решает два DEFERRED, заходящих в Gate 5: #6 (семантика
  бюджета поисков per-run vs per-loop) и #7 (graceful-обработка падения прогона).
  Три решения фиксируются одним D-13 — они переплетены (см. «Переплетение» ниже).

**D-13.1 — Бюджет: per-loop (один `ResearchContext` на весь запрос).**
- Решение: единый `ResearchContext` создаётся один раз на весь запрос и переиспользуется
  на всех доработках петли, а НЕ пересоздаётся свежим на каждую итерацию доработки.
- Причина: только так предохранитель D-03 честен. При per-run (свежий контекст на
  доработку) худший случай — `(1 + MAX_ROUNDS) × DEFAULT_SEARCH_BUDGET = 36` поисков
  на запрос против обещанных `12`, а промпт менеджера «budget for the ENTIRE run»
  на уровне системы врёт. Механика подъёма контекста (2b/2d) уже написана.
- Отклонён per-run. Тень отклонённого: доработка тратила бы бюджет втрое,
  предохранитель лжёт обещанному лимиту.
- Названная тень принятого (per-loop): при пустом остатке бюджета доработка НЕ
  может добыть новые данные — поиска нет, бюджет исчерпан. Поэтому `missing_topics`,
  требующие новых фактов, на такой итерации структурно закрыть нечем; улучшение
  возможно только переписыванием уже собранного. Разведено явно: невозможность
  новых данных — **безусловна** (следствие исчерпанного бюджета); холостая
  итерация — **возможный исход** (если весь дефицит был именно в новых фактах).
  Принята осознанно; видна в логе (счётчик поисков не растёт при росте раунда).
- Закрывает DEFERRED #6.

**D-13.2 — Канал наружу: петля → `AsyncIterator[ProgressEvent]` (генератор).**
- Решение: слой петли отдаёт наружу асинхронный генератор событий прогресса
  `ProgressEvent` (`stage` / `payload` / `report`). Поток ВСЕГДА завершается ровно
  одним терминальным событием `final`, несущим **непустой (обязательно заполненный)**
  `report: ReportData`. Инвариант: `report` у `final` НЕ может быть `None` — за это
  цепляется E2E (`isinstance(report, ReportData)`). Этим сохраняется E2E-контракт
  (D-06/D-12): труба по-прежнему собирает `ReportData`.
- Причина: критерий Gate 5 «накопительный лог + счётчики» требует живого стриминга
  промежуточных оценок; Gradio стримит именно через генераторы. Callback лёг бы в
  UI через мост-очередь (callback → queue → генератор) = вскрытие контракта посреди
  вёрстки экранов.
- Отклонён callback-контракт (`on_iteration(score, feedback)` как публичный интерфейс
  наружу). Цена принятого: правка уже принятых 4b (петля) + 4c (E2E) — слой петли
  переводится с колбэка на генератор.
- Обслуживает критерий Gate 5 «промежуточные результаты оценщика видны».
- Уточнение семантики терминалов (по итогу реализации 5-pre-c; вариант 1 ПРИНЯТ
  владельцем). Реализация обнажила случай, которого исходный текст D-13.2/D-13.3 не
  предусматривал: сбой прогона (`MaxTurnsExceeded`/сеть/API) ПОСЛЕ того, как хотя бы
  одна версия отчёта уже оценена (падение на доработке при уже готовой первой версии).
  Это гибрид «сбой прогона + валидный отчёт на руках». Достройка контракта (НЕ смена
  решения — исходный текст этот путь просто не описывал):
  * `final` = терминал «есть оценённый непустой `report: ReportData`». Достигается
    ДВУМЯ путями: **(а)** нормальный выход — порог взят ИЛИ лимит итераций исчерпан —
    чистый успех; **(б)** сбой прогона ПОСЛЕ хотя бы одной оценённой версии — graceful
    exit ЛУЧШЕЙ по score (та же ветка, что исчерпание лимита: наружу неидеальный, но
    валидный отчёт).
  * `payload["graceful"]: bool` — записанный контракт флага: ОТСУТСТВУЕТ (трактуется
    как `False`) на пути (а), `True` на пути (б). Потребитель читает через
    `payload.get("graceful", False)`; UI может показать маркер «получено после сбоя»,
    НЕ блокируя годный отчёт.
  * Граница `final` vs `failed` — по НАЛИЧИЮ оценённой версии, НЕ по тяжести сбоя:
    есть хоть одна оценённая версия → `final` (при сбое — graceful); нет ни одной
    (падение до первой оценки) → `failed`, `report=None`, сообщение в
    `payload["message"]` (D-13.3).
  * Дренер `run_loop_to_report` (5-pre-c) флаг `graceful` СОЗНАТЕЛЬНО игнорирует:
    любой `final` = успех, отдаёт `report` (годный отчёт есть отчёт); исключение
    поднимает только на `failed`. Различение нормального/graceful-финала — забота
    потребителя, читающего поток напрямую (UI), а не дренера.

**D-13.3 — Падение прогона: терминал `failed`.**
- Решение: петля оборачивает вызов ядра (Зона 2 / прогон) в `try/except`. Если к
  моменту падения ЕСТЬ хотя бы одна оценённая версия отчёта → graceful exit лучшей
  по score (как при исчерпании лимита итераций, 4b). Если оценённой версии НЕТ →
  генератор эмитит терминальное событие `stage="failed"` вместо `final`.
- Форма двух терминалов (взаимоисключающие, поток завершается ровно одним):
  * `final` — несёт непустой `report: ReportData` (не `None`; D-13.2), `payload` про
    итог;
  * `failed` — `report` ОТСУТСТВУЕТ (`None`), несёт человекочитаемое сообщение об
    ошибке в `payload`.
  E2E на `failed` `report` НЕ ждёт и `isinstance` НЕ проверяет (проверка `isinstance`
  цепляется только к `final`, D-13.2).
- Причина: без этого падение прогона (`MaxTurnsExceeded` / сеть / API) = исключение
  улетает наверх → на Space зависший экран. При канале-генераторе (D-13.2) `failed` —
  просто ещё один тип терминального события, дешевле заложить сразу: иначе контракт
  потока правится дважды (урок 4d — контракт потока правится один раз).
- Зависит от D-13.2 (генератор как несущий канал).
- Закрывает DEFERRED #7.

**Переплетение (зафиксировано).** Подъём `ResearchContext` из `run_research`
наружу обслуживает сразу и бюджет (D-13.1 — общий контекст на весь запрос), и
счётчик поисков для UI (число поисков в лог). Генератор (D-13.2) несёт наружу
`score`/`feedback` + счётчик + терминал `failed` (D-13.3). Поэтому три решения —
один пакет, не три независимых правки.

- Последствия / карта под-шагов 5.0: 5-pre-a (эта запись в DECISIONS) → 5-pre-b
  (подъём `ResearchContext`) → 5-pre-c (петля → генератор + `failed`) → 5-pre-d
  (E2E под поток). Вёрстка экранов Gate 5 — ПОСЛЕ закрытия 5.0.
