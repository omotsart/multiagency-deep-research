# WORKFLOW.md — процесс разработки Deep Research

> Это документ о том, **как мы строим приложение** (процесс сборки), а не о том,
> как приложение работает в runtime. Runtime-архитектура описана в `CLAUDE.md`
> и `DECISIONS.md`. Не путать два слоя.

## Как читать этот файл

Разработка разбита на **gates** (вехи). Gate нельзя пометить закрытым, пока не
выполнены все его критерии. Крупные gates разбиты на **под-gates** (атомарные
задачи). Вход и выход каждого gate — через документ `handover/gate-N.md`
(см. `HANDOVER.md`).

Статусы: `TODO` · `IN PROGRESS` · `BLOCKED` · `DONE` · (задача) `DEFERRED`.

Правила процесса (замер контекста, deferred, право вето) — в `RULES.md`.
Все зафиксированные архитектурные решения — в `DECISIONS.md`.

E2E-контракт (happy-path на моках) закладывается на Gate 2, оживает на Gate 3,
держится зелёным до конца. Ломается E2E → работа стоп, чиним контракт.

## Gate 1 — Clarifier + обогащение брифа

**Цель.** Первый агент задаёт 3 уточняющих вопроса; ответы склеиваются с темой
в единый бриф. Живёт ВНЕ автономного ядра (человек в цикле).

**Критерии закрытия:**
- [x] `clarifier_agent` с `output_type` на ровно 3 вопроса (Pydantic).
- [x] Функция склейки `тема + Q&A → brief` (детерминированная, чистая).
- [x] Юнит-тест склейки брифа (контракт передачи данных).
- [x] Юнит-тест валидности Pydantic-схемы вопросов.

**Тесты:** unit (склейка брифа, схема).
**Не тестируем:** «умность»/качество самих вопросов — оценивает человек.

## Gate 3 — Автономный менеджер

**Цель.** Менеджер-агент оркестрирует инструменты сам (agents-as-tools),
принимает решения о числе и направлении поисков, включая динамическую вторую волну.

**Критерии закрытия:**
- [x] `manager_agent` с инструментами: план, поиск, писатель.
- [x] Промпт менеджера: бюджет поисков, инструкция про вторую волну по пробелам.
- [x] E2E happy-path ЗЕЛЁНЫЙ: `бриф → план → поиски → отчёт` на моках.
- [x] Юнит-тест: менеджер уважает бюджет поисков (не превышает лимит).

**Тесты:** unit (соблюдение бюджета) + E2E (контракт трубы).
**Не тестируем:** конкретные решения LLM о запросах — недетерминированны.

## Gate 4 — Evaluator-Optimizer петля

**Цель.** Оценщик проверяет отчёт; петля дорабатывает до порога либо до лимита.
Петля — детерминированный `while` в коде (НЕ отдана менеджеру), ради гарантии лимита.

**Критерии закрытия:**
- [x] `evaluator_agent` с `output_type`: `score`, `passed`, `feedback`, `missing_topics`. _(4a)_
- [x] Петля: `while not passed and rounds < MAX_ROUNDS` (`MAX_ROUNDS = 2`). _(4b)_
- [x] Порог `score >= 7` (не 10). _(4b, в коде: `passed = score >= SCORE_THRESHOLD`)_
- [x] Graceful exit: лимит исчерпан → лучшая версия по score наружу. _(4b)_
- [x] Каждая итерация отдаёт наружу `score` и `feedback` (для UI). _(4b, колбэк `on_iteration`)_
- [x] Юнит-тесты: выход по порогу; выход по лимиту; выбор лучшей версии. _(4b, +тай-брейк)_
- [x] E2E расширен: труба проходит через оценщик. _(4c, два сценария)_

**Тесты:** unit (сценарии петли + схема/стык финализатора) + E2E (труба с оценщиком).
**Финал (D-09, 4d):** после петли — единственный handoff на `finalizer_agent` через
агент-шлюз (`research_and_finalize`); финализатор — LLM-агент, оформляет выдачу.

## Gate 5 — UI (три экрана)

**Цель.** Gradio-интерфейс: тема → вопросы → прогресс+отчёт. Переключение
через `visible` + `gr.State`. Лог + счётчики. Промежуточные результаты оценщика видны.

**Критерии закрытия:**
- [x] Три группы компонентов, переключение видимости через `gr.State`. _(5a)_
- [x] Экран 2: 3 статичных текстбокса (Подход А, без `@gr.render`), email-галочка. _(5a)_
- [x] Экран 3: накопительный лог + счётчики (поисков / итераций / score). _(5b)_
- [x] Счётчик версий «Версия N / M»: `N = round + 1` (текущая в работе), `M =
      MAX_ROUNDS + 1` (из константы, не хардкод); терминология «версия» сквозная (D-14.1). _(5b)_
- [x] Три плашки терминала D-13: `final` чистый → зелёная «Завершено»; порог не взят
      (graceful ИЛИ `score < SCORE_THRESHOLD`) → жёлтая (score лучшей + причина); `failed` →
      красная (сообщение из `payload["message"]`). Кнопка «Новое исследование» — на
      ЛЮБОМ терминале, включая failed и жёлтый (D-14.2, правило плашки уточнено на 5b —
      score-based). _(5b)_
- [x] Провокатор всех исходов — в приёмочной обвязке (`acceptance_provocateur.py`, env
      `MAD_FORCE_TERMINAL`), НЕ элемент живого UI (D-14.3); четыре режима
      (final/limit/graceful/failed). _(5b)_
- [x] Свёрнутые экраны 1–2 — резюме-строки без «Изменить»/редактирования во время
      прогона; сброс только «Новое исследование» на терминале, очищает поля (D-14.4). _(5a/5b)_
- [x] Кастомная тема + CSS только через свои `elem_classes` (безопасно); тема sky база,
      переключатель светлая/тёмная. _(5a/5b)_
- [x] Ручная проверка флоу от начала до конца, **включая ПРОГОН ВСЕХ ТРЁХ исходов**
      (`final` чистый / жёлтый graceful+limit / `failed`) — каждый терминал вызван руками,
      плашка + сброс проверены; узкий экран и стриминг проверены (D-14.2/D-14.3). _(владелец)_

**Тесты:** ручная проверка (автотесты Gradio-фронта не пишем — не стоит усилий).
**Приёмку подтвердил владелец** (happy-path, 4 терминала, кнопки, тема, скачивание,
сброс). Детерминированная обвязка слоя петли (генератор/дренер/failed/бюджет) —
покрыта юнит/E2E на под-гейте 5.0 (54 passed).
