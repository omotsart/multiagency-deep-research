## Цель gate
Gradio-интерфейс из трёх экранов: **тема → уточняющие вопросы → прогресс+отчёт**.
Переключение экранов через `visible` + `gr.State`. На экране прогресса —
накопительный лог + счётчики (поисков / итераций / score); промежуточные
результаты оценщика видны. Email-галочка как демонстрация (D-05, без реальной
отправки).

## Gate 5 открывается под-гейтом 5.0 — контракт слоя петли (D-13) ДО экранов
Вёрстка экранов НЕ начинается, пока не закрыт **под-гейт 5.0** — контракт слоя
петли под UI. Он ложится ПОД экраны: определяет, как петля отдаёт прогресс наружу
(генератор), как считается бюджет на весь запрос (per-loop) и как поток завершается
при падении (`failed`). Класть экраны поверх неопределённого контракта = вскрытие
контракта посреди вёрстки (урок 4d — контракт потока правится один раз).

**D-13 (ПРИНЯТО, 2026-07-03, владелец подтвердил)** — три решения одним пакетом:
- **D-13.1** — бюджет per-loop: один `ResearchContext` на весь запрос (не свежий на
  доработку). Честный предохранитель D-03. **Закрывает DEFERRED #6.**
- **D-13.2** — канал-генератор: петля → `AsyncIterator[ProgressEvent]`
  (`stage`/`payload`/`report`), поток всегда завершается одним `final` с `ReportData`
  (сохраняет E2E-контракт D-06/D-12). Отклонён callback.
- **D-13.3** — failed-терминал: `try/except` вокруг ядра; есть оценённая версия →
  graceful exit лучшей по score; нет → событие `stage="failed"`. Зависит от D-13.2.
  **Закрывает DEFERRED #7.**

## Сделано в этой сессии
- **5-pre-a — DONE (ждёт приёмки владельца).** D-13 записан в DECISIONS.md
  (статус ПРИНЯТО, дата 2026-07-03, владелец подтвердил) — три под-пункта
  (13.1 per-loop / 13.2 генератор / 13.3 failed) с причинами, отклонёнными
  вариантами, названной тенью принятого per-loop и зафиксированным переплетением
  (подъём контекста обслуживает и бюджет, и счётчик UI; генератор несёт
  score/feedback + счётчик + failed). Открыт этот handover. **Кода не трогали.**
- **5-pre-b — DONE (ждёт приёмки владельца).** `src/run_research.py`: `run_research`
  получил необязательный параметр `context: ResearchContext | None = None`. Внутри
  `ctx = context if context is not None else ResearchContext()`, дальше
  `Runner.run(..., context=ctx, ...)` как раньше. Дефолт НЕ изменён — вызов без
  аргумента создаёт свежий контекст (полная обратная совместимость; существующие
  тесты держатся БЕЗ правок). Цель — дать вызывающему (будущей петле-генератору
  5-pre-c) передать ОДИН `ResearchContext` на весь запрос: техническая основа
  per-loop (D-13.1) и подъёма счётчика поисков наружу для UI. Сам per-loop здесь
  ещё НЕ включён — только возможность в сигнатуре ядра; петля/менеджер/предохранители/
  E2E НЕ тронуты. Новый юнит-тест `tests/test_run_research_context.py`: переданный
  `context` используется прогоном (identity `is`), поиски списываются в НЕГО, а не
  в свежий (на моках, без ключа).
- **5-pre-c — DONE (ждёт приёмки владельца).** Сердце D-13 — петля переоформлена из
  колбэка в генератор, включён per-loop, заложен failed-терминал. `src/evaluator_loop.py`:
  * **И-1 генератор (D-13.2).** `research_with_evaluation` теперь чистый
    `async def -> AsyncIterator[ProgressEvent]`. `ProgressEvent(stage, payload, report)`
    (frozen dataclass). Стадии: `research_started` / `draft_ready` / `evaluation`
    (payload `round`/`score`/`feedback`/`missing_topics`/`searches_used`) /
    `revision_started` / `final` / `failed`. Промежуточные события `report=None`.
    Колбэк `on_iteration`/`ProgressCallback`/`_emit` УДАЛЁН — роль взял поток событий.
  * **И-2 терминал (D-13.2).** Поток ВСЕГДА кончается ровно одним терминалом:
    `final` с НЕПУСТЫМ `report: ReportData` (не None — за это цепляется E2E
    `isinstance`). `final`/`failed` взаимоисключающи.
  * **И-3 per-loop (D-13.1).** ОДИН `ResearchContext()` создаётся на весь запрос и
    передаётся в КАЖДЫЙ `run_research(brief, context=ctx)` — и первый прогон, и
    каждую доработку (проверено глазами: доработка получает ТОТ ЖЕ `ctx`).
    `ctx.searches_used` идёт в payload (задел под счётчик UI).
  * **И-4 failed (D-13.3).** Тело обёрнуто в `try/except`: при падении есть
    оценённая версия → graceful exit `final` с лучшей по score (та же ветка, что
    лимит); нет ни одной → терминал `failed` (`report=None`, сообщение в
    `payload["message"]`).
  * **Старые свойства сохранены:** порог `>= 7` (SCORE_THRESHOLD), `MAX_ROUNDS=2`,
    `while not passed and rounds < MAX_ROUNDS`, graceful exit возвращает ЛУЧШУЮ по
    score, тай-брейк `>` (ничья → ранняя). Тесты проверяют ИХ ЖЕ.
  * **Дренер `run_loop_to_report(brief) -> ReportData`** (вариант C, ПРИНЯТ
    владельцем): каноническая свёртка потока к отчёту — `final` → `report`, `failed`
    → исключение с сообщением из payload (НЕ None). **Живёт в `evaluator_loop.py`**
    рядом с генератором и контрактом стадий (обоснование: свёртка ЭТОГО потока, знает
    про терминалы; отдельный модуль был бы слоем без необходимости, RULES §6). Его
    зовут finalizer (сейчас), E2E (5-pre-d), UI (вёрстка) — знание «как достать report
    из потока» в ОДНОЙ точке.
  * **finalizer-стык поправлен СЕЙЧАС (в границах 5-pre-c, раз конверсия его
    затронула — поднято владельцу, вариант C принят).** `src/finalizer_agent.py`:
    `research_and_finalize` зовёт `run_loop_to_report(brief)` вместо
    `research_with_evaluation(brief)` (своп импорта + вызова + докстринг). Тесты
    стыка 4d остались зелёными (финализатор получает тот же `ReportData`, просто
    через дренер).
  * Тесты `tests/test_evaluator_loop.py` переписаны под генератор: 4 старых сценария
    (порог / лимит / лучшая-по-score / тай-брейк) проверяют ТЕ ЖЕ свойства через
    итерацию потока + терминал; +`failed`-терминал (падение без оценённой версии
    → ровно один `failed` без report); +graceful-`final` (падение ПОСЛЕ первой оценки
    → ровно один `final` с непустым отчётом + `payload["graceful"] is True`, `failed`
    в потоке нет); +2 теста дренера (happy → ReportData; failed → исключение, не None).
    Хелпер `_terminal` держит И-2 (ровно один терминал, последний).
  * **Уточнение семантики терминалов (по итогу 5-pre-c, вариант 1 ПРИНЯТ владельцем).**
    Реализация обнажила гибрид «сбой прогона + валидный отчёт на руках» (падение на
    доработке при уже оценённой первой версии). Достройка D-13.2 (не смена решения):
    `final` = «есть оценённый непустой report» двумя путями — (а) нормальный выход,
    (б) graceful после сбоя; `payload["graceful"]: bool` — записанный контракт флага
    (отсутствует/False → путь а, True → путь б); граница `final`/`failed` — по НАЛИЧИЮ
    оценённой версии, не по тяжести сбоя. Дренер флаг СОЗНАТЕЛЬНО игнорит (любой
    `final` = успех) — комментарий в дренере со ссылкой на D-13.2. Код эмиссии
    `final`/`failed` НЕ менялся — уже реализовывал вариант 1; правки 5-pre-c-addendum:
    запись контракта (DECISIONS) + покрытие (graceful-тест) + комментарий дренера.
  * **НЕ тронуты:** менеджер/предохранители/ядро (`run_research` готов с 5-pre-b),
    E2E (5-pre-d), сам механизм handoff-шлюза.
