## Цель gate
Gradio-интерфейс из трёх экранов: **тема → уточняющие вопросы → прогресс+отчёт**.
Переключение экранов через `visible` + `gr.State`. На экране прогресса —
накопительный лог + счётчики (поисков / итераций / score); промежуточные
результаты оценщика видны. Email-галочка как демонстрация (D-05, без реальной
отправки).

## Gate 5 открывается под-гейтом 5.0 — контракт слоя петли (D-13) ДО экранов
Вёрстка экранов НЕ начинается, пока не закрыт **под-гейт 5.0** — контракт слоя
петли под UI. Он ложится ПОД экраны: определяет, как петля отдаёт прогресс наружу
(генератор), как считается бюджет на весь запрос (per-loop) и как поток завершается
при падении (`failed`). Класть экраны поверх неопределённого контракта = вскрытие
контракта посреди вёрстки (урок 4d — контракт потока правится один раз).

**D-13 (ПРИНЯТО, 2026-07-03, владелец подтвердил)** — три решения одним пакетом:
- **D-13.1** — бюджет per-loop: один `ResearchContext` на весь запрос (не свежий на
  доработку). Честный предохранитель D-03. **Закрывает DEFERRED #6.**
- **D-13.2** — канал-генератор: петля → `AsyncIterator[ProgressEvent]`
  (`stage`/`payload`/`report`), поток всегда завершается одним `final` с `ReportData`
  (сохраняет E2E-контракт D-06/D-12). Отклонён callback.
- **D-13.3** — failed-терминал: `try/except` вокруг ядра; есть оценённая версия →
  graceful exit лучшей по score; нет → событие `stage="failed"`. Зависит от D-13.2.
  **Закрывает DEFERRED #7.**

## Вход 5b — обязательные правки к D-14.1 (всплыли при ревью макета)
На 5a лога ещё нет — здесь только ЗАПИСАНО, не реализовано. При наполнении экрана 3
(5b) обязательно:
1. **Терминология «версия/черновик» относится к АРТЕФАКТУ-ОТЧЁТУ, не к волнам поиска.**
   Волны поиска сохраняют свой словарь: в логе «волна N: X/Y сводки», а НЕ «версия N».
   Иначе лента противоречива (счётчик «Версия N» существовал бы до готовности версии).
   Счётчик версий N/M (D-14.1) считает именно версии отчёта (`round + 1`), поиски в
   логе — своим словарём.
2. **Пара терминов «версия» (артефакт) + «доработка» (процесс) — осознанная,**
   зафиксировать в UI-копирайтинге: версия — что получилось (артефакт-отчёт),
   доработка — что делается между версиями (процесс оценки→ревизии).

**Чек-лист ручной приёмки 5b (добавить к прогону):**
- проверка на УЗКОМ экране: колонки складываются, лог не должен «хоронить» отчёт
  (возможно `gr.Accordion` для лога);
- **autoscroll лога** — лента прилипает к низу по мере поступления событий;
- **таймер прогона** — «Идёт исследование · M:SS»;
- **стриминг как увидит Spaces**, а не только `python app.py`: прогнать без
  `inbrowser`, с явной `.queue()`. Генераторы идут разными транспортами — локальный
  `.queue()` на Windows vs SSE на Spaces; классический симптом рассинхрона — лог живой
  локально, но приходит ПАЧКОЙ в конце на Spaces. Полная проверка живости лога на
  настоящем Space — Gate 6 (**DEFERRED #9**, зазор среды Windows→Spaces).
