## Что это за проект

Автономный агент-исследователь на **OpenAI Agents SDK** с интерфейсом на
**Gradio**. Пользователь задаёт тему → агент уточняет детали → автономно
исследует → пишет отчёт с самопроверкой качества → показывает результат.

Проект развивает учебный базовый пример (последовательный вызов агентов) в
сторону настоящей агентной системы с автономией и контролем качества.

## Стек

- Python, OpenAI Agents SDK (`agents`), Pydantic (структурированный вывод).
- Gradio (UI) — версия **зафиксирована** в `requirements.txt`.
- `asyncio` — параллельные поиски.
- Трейсинг — встроенный в SDK.

## Runtime-архитектура (три зоны)

**Зона 1 — Прелюдия (человек в цикле, ВНЕ автономии).**
`clarifier_agent` задаёт 3 вопроса → UI ждёт ответы → склейка `тема + Q&A → brief`.
Бриф — единый источник намерения для всего пайплайна.

**Зона 2 — Автономное ядро (решает LLM).**
`manager_agent` вызывает специалистов как инструменты (`.as_tool()`):
планировщик, параллельный поиск, писатель. Сам решает число и направление
поисков, включая динамическую вторую волну по пробелам в данных.
Паттерн — **agents-as-tools** (менеджер владеет финальным ответом), НЕ handoffs.

**Зона 3 — Контроль качества (детерминированный код).**
`evaluator_agent` оценивает отчёт (`score`, `passed`, `feedback`, `missing_topics`).
Петля дорабатывает в `while`-цикле в Python до порога `score >= 7` либо до
`MAX_ROUNDS = 2`. Петля — в коде, НЕ у менеджера, ради гарантии лимита.
После утверждения отчёта менеджер делает единственный handoff на `finalizer_agent`,
оформляющий финальную выдачу (см. D-09).

## Предохранители автономности (критично)

- `max_turns` в `Runner.run()` — потолок ходов агента.
- Бюджет поисков в `RunContextWrapper` — поисковый tool отказывает при исчерпании.
- Промпт менеджера явно задаёт бюджет.
- Evaluator-петля: жёсткий `MAX_ROUNDS`, порог вместо перфекционизма, graceful exit.

## Модели

Более способная модель — для менеджера (принимает решения). `gpt-4o-mini` —
для простых поисковых сводок. (Точные строки моделей — в коде/конфиге.)

## Структура файлов (исходный базовый пример)

- `deep_research.py` — Gradio UI (станет/переедет в `app.py`).
- `research_manager.py` — оркестратор (эволюционирует в автономный менеджер).
- `planner_agent.py` — планировщик поисков → `WebSearchPlan`.
- `search_agent.py` — поиск + сводка (`WebSearchTool`).
- `writer_agent.py` — синтез отчёта → `ReportData`.
- `email_agent.py` — отправка (в деплое: галочка без реальной отправки).

Новое по ходу разработки: `clarifier_agent`, `evaluator_agent`, `manager_agent`.

## Как запускать

- Локально: `.env` с `OPENAI_API_KEY`, затем `python app.py`.
- Тесты: `pytest` (моки LLM, без реальных вызовов и без ключа).

## Что НЕ делаем (антипаттерны для этого проекта)

- Не используем handoffs в основном конвейере (agents-as-tools). Исключение — один
  изолированный handoff на финальном оформлении отчёта (см. D-09).
- Не используем `@gr.render` для вопросов (риск «перекошенного» UI — Подход А).
- Не тестируем автоматически: качество текста, «умность» вопросов, Gradio-фронт.
- Не коммитим секреты. Не хардкодим email-адреса.
