# RULES.md — жёсткие правила проекта

> Инварианты. Нарушение любого — стоп-сигнал. Правила меняются только через
> запись в `DECISIONS.md` с явным «принято» от владельца.

## 1. Право вето владельца

- Критические решения (архитектура, смена паттерна, изменение контракта данных,
  границы gate) вносит **только владелец проекта**.
- Claude предлагает изменения, помечает их `ПРЕДЛОЖЕНО`, но НЕ фиксирует как
  принятые и НЕ реализует до явного «принято».
- Всё принятое — в `DECISIONS.md`. Роль `gate-reviewer` проверяет, что ни одно
  критическое изменение не просочилось мимо журнала.

## 2. Управление контекстом сессии

- Каждая сессия **начинается с замера** стартового объёма контекста.
- Порог предупреждения — **150K токенов**. При достижении И незакрытом gate:
  Claude обязан остановиться и выписать `handover/gate-N.md` для переноса в новую
  сессию (не продолжать впритык).
- Жёсткий потолок — **170K токенов**. За него не заходим.
- Буфер 150K→170K зарезервирован под написание самого handover.

## 3. Gates и deferred

- Gate/под-gate помечается `DONE` только когда **все** его критерии выполнены.
- Вопрос, возникший по ходу, но вне scope текущего gate, **не решается на месте**.
  Он записывается в реестр DEFERRED в `WORKFLOW.md` с привязкой к будущему gate и
  приносится туда на обсуждение.
- Вход и выход каждого gate — через документ handover (см. `HANDOVER.md`).

## 4. Тестовая дисциплина

- Тестируем **детерминированную обвязку**: предохранители, склейку брифа,
  Pydantic-схемы, evaluator-петлю, E2E-контракт трубы.
- НЕ тестируем автоматически: качество текста, «умность» вопросов, Gradio-фронт.
- Тесты — на **моках** LLM. Без реальных вызовов, без ключа в CI.
- E2E зелёный — инвариант с Gate 3. Красный E2E = работа стоп, чиним контракт.

## 5. Безопасность и гигиена

- Секреты (`OPENAI_API_KEY` и пр.) — только через окружение / Secrets на Spaces.
  Никогда не в коде и не в репозитории.
- Захардкоженные email-адреса запрещены.
- `.env` — в `.gitignore`. В репозитории только `.env.example`.
- Версия Gradio зафиксирована (страховка от «перекошенного» UI на новой версии).

## 6. Границы простоты (анти-переусложнение)

- Проект небольшой. Не добавляем архитектурных слоёв без явной необходимости.
- Не заводим handoffs, `@gr.render`, лишние ролевые файлы, Skills — пока
  конкретная потребность не зафиксирована в `DECISIONS.md`.
