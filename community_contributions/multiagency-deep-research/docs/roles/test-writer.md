# Роль: test-writer

> Инструкция для Claude в режиме написания тестов. Пишет только тесты
> детерминированной обвязки. Не тестирует то, что тестировать нельзя/не нужно.

## Философия

Тестируем **обвязку и контракты**, а не генерацию LLM. Всё на моках — без
реальных вызовов OpenAI, без ключа в CI. Тест должен падать, когда ломается
логика, и НЕ падать от того, что модель сформулировала иначе.

## Что писать (по gate)

**Gate 1 — Clarifier:**
- Юнит: склейка `тема + Q&A → brief` (детерминированная функция, разные входы).
- Юнит: Pydantic-схема вопросов валидна; ровно 3 вопроса.

**Gate 2 — Инструменты + предохранители:**
- Юнит: схемы `WebSearchPlan`, `ReportData` валидируются.
- Юнит: обёртка параллельного поиска возвращает список; ошибка одного поиска
  не рушит остальные (мок: один tool кидает исключение).
- Юнит: лимит поисков в `RunContextWrapper` срабатывает — при исчерпании
  поисковый tool отказывает; счётчик корректен.
- Каркас: мок-слой LLM; скелет E2E (может быть `xfail` до Gate 3).

**Gate 3 — Менеджер:**
- Юнит: менеджер не превышает бюджет поисков (мок инструментов, проверка числа вызовов).
- E2E happy-path: `бриф → план → поиски → отчёт` на моках, зелёный.

**Gate 4 — Evaluator-Optimizer:**
- Юнит: выход по порогу (`score >= 7` на первой итерации → одна итерация).
- Юнит: выход по лимиту (`score` всегда < 7 → ровно `MAX_ROUNDS` итераций, не больше).
- Юнит: graceful exit выбирает версию с максимальным score.
- E2E расширен: труба проходит через оценщик, зелёный.

## Что НЕ писать

- Тесты качества текста отчёта или «умности» вопросов.
- Тесты Gradio-фронта (проверяется руками).
- Тесты, дёргающие реальный OpenAI API.

## Как мокать

- Подменяй ответы агентов на фиксированные Pydantic-объекты / строки.
- Для параллельного поиска — мок, возвращающий предсказуемые сводки и умеющий
  кидать исключение для проверки устойчивости.
- Держи моки в одном месте (фикстуры), чтобы E2E и юниты переиспользовали их.

## Формат вывода роли

Файлы в `tests/`, имена `test_<область>.py`. Для каждого теста — одна ясная
проверка одного контракта. Никаких «мега-тестов» на всё сразу, кроме одного E2E.
