## Цель gate
Оценщик проверяет готовый отчёт против брифа; петля дорабатывает отчёт до порога
качества либо до лимита итераций. Петля — детерминированный `while` в КОДЕ (НЕ
отдана менеджеру-агенту, D-02), ради гарантии лимита итераций.

## Критерии закрытия (чек-лист дословно из WORKFLOW.md, секция Gate 4)
- [x] `evaluator_agent` с `output_type`: `score`, `passed`, `feedback`, `missing_topics`. _(4a)_
- [x] Петля: `while not passed and rounds < MAX_ROUNDS` (`MAX_ROUNDS = 2`). _(4b)_
- [x] Порог `score >= 7` (не 10). _(4b)_
- [x] Graceful exit: лимит исчерпан → лучшая версия по score наружу. _(4b)_
- [x] Каждая итерация отдаёт наружу `score` и `feedback` (для UI). _(4b)_
- [x] Юнит-тесты: выход по порогу; выход по лимиту; выбор лучшей версии. _(4b)_
- [x] E2E расширен: труба проходит через оценщик. _(4c)_

**Тесты:** unit (три сценария петли) + E2E (труба с оценщиком).
**Не тестируем:** «умность» самих оценок LLM — недетерминированны (RULES §4).

## Карта под-шагов Gate 4 (атомарная приёмка по каждому, владелец принимает каждый)
- **4a — `evaluator_agent` + схема** (`score`/`passed`/`feedback`/`missing_topics`).
  ← **этот шаг.** Только сам агент со схемой и промптом. Без петли, без рантайма,
  без E2E, без finalizer.
- **4b — Петля в КОДЕ:** детерминированный `while not passed and rounds <
  MAX_ROUNDS` (`MAX_ROUNDS = 2`), порог `score >= 7`, graceful exit (лучшая версия
  по score наружу при исчерпании лимита), выбор лучшей версии. Каждая итерация
  отдаёт наружу `score`/`feedback` (задел под UI). + юнит-тесты (выход по порогу /
  по лимиту / выбор лучшей версии). **Петля живёт в Python, НЕ у менеджера (D-02):
  не отдавать петлю качества менеджеру как инструмент, даже если проще — гарантия
  лимита держится детерминированным кодом.**
- **4c — Расширение E2E:** труба проходит через оценщик, остаётся зелёной.
- **4d — Стык D-09 / finalizer:** после утверждения отчёта петлёй по D-09/D-12
  ложится отложенный **handoff на `finalizer_agent`**. Где именно ложится граница
  handoff и входит ли в Gate 4 — **решение владельца (RULES §1)**, приносится при
  постройке петли; Claude Code НЕ решает сам.

## Сделано в этой сессии
- **4a — DONE (ждёт приёмки владельца).** `src/evaluator_agent.py`: агент
  `evaluator_agent` со схемой `EvaluationFeedback` (Pydantic `output_type`), поля
  ровно `score` (0-10, порог закрытия петли >= 7 — в докстринге/Field), `passed`
  (bool), `feedback` (str — что улучшить), `missing_topics` (list[str] — пробелы).
  Промпт: оценивает готовый отчёт против исходного брифа, возвращает
  структурированную оценку; `feedback`/`missing_topics` явно описаны как
  руководство к доработке (их потребит петля на 4b). Порог/логика петли в промпт
  НЕ зашиты — петля живёт в коде (D-02), оценщик только оценивает. Модель `gpt-4o`
  (как менеджер): оценка — содержательное суждение, не сводка (обоснование по
  CLAUDE.md). Стиль как у `planner_agent.py`/`writer_agent.py` (Agent + Pydantic).
  НЕ создана петля, НЕ подключён рантайм, НЕ тронут E2E, НЕ создан finalizer.
- **4b — DONE (ждёт приёмки владельца).** `src/evaluator_loop.py`: петля
  Evaluator-Optimizer — детерминированный `while not passed and rounds <
  MAX_ROUNDS` в КОДЕ (D-02), НЕ инструмент менеджера. `research_with_evaluation(
  brief, on_iteration=None) -> ReportData`: Зона 2 (`run_research`) готовит отчёт,
  оценщик судит против брифа, feedback/missing_topics скармливаются обратно в
  доработку (обогащённый бриф → пере-прогон Зоны 2). Константы `SCORE_THRESHOLD=7`,
  `MAX_ROUNDS=2`. **Пороговое решение — в коде:** `passed = score >= SCORE_THRESHOLD`;
  поле `EvaluationFeedback.passed` от LLM остаётся советом для UI, управляющий
  инвариант — код-порог (гарантия D-02, критерий «score >= 7, не 10»). Graceful
  exit — наружу лучшая по score версия (ничьи оставляют раннюю). Колбэк
  `on_iteration(score, feedback)` эмитит наружу каждую оценку (задел под UI, Gate 5).
  Три юнит-теста (`tests/test_evaluator_loop.py`): выход по порогу / по лимиту /
  выбор лучшей версии (+ тай-брейк равных score, добавка при приёмке 4b). НЕ тронут
  E2E (4c), НЕ создан finalizer/handoff (4d).
- **4c — DONE (ждёт приёмки владельца).** E2E расширен через оценщик — ДВА сценария
  в `tests/test_e2e_pipeline.py`, труба идёт через `research_with_evaluation` (Зона 3,
  evaluator-петля поверх Зоны 2), а не голый `run_research`.
  * **Happy-path** (`test_e2e_happy_path_brief_to_report`): проходная оценка
    (`score=8 >= 7`) на первой итерации → без доработок. Контракт: результат —
    `ReportData`, труба прошла план→поиск→писатель→**оценщик** (`.ran(...)`).
  * **С доработкой** (`test_e2e_revision_path_fail_then_pass`, добавлен при приёмке
    4c): оценка `5` (ниже порога) → `_revision_brief` → повторный `run_research` →
    оценка `8` (порог) → выход. Проверка: менеджер и оценщик отработали ПО ДВА раза
    (`calls.count(...) == 2`) — повторный прогон реально собрался, а не заглушён.
    Гоняет multi-round путь сценарной модели (свежий `_step=0` на диспатч), ранее
    подтверждённый только рассуждением.
  Мок-слой `tests/_mocks.py` достроен (на 4c): билдер `make_evaluation` (дефолт
  проходной), диспетчер отдаёт канон `EvaluatorAgent → EvaluationFeedback` по имени
  агента, `configure(evaluation=...)`. Последовательность оценок 5→8 во втором тесте
  — **без новой инфраструктуры мок-слоя**: тонкая тест-локальная обёртка над готовым
  `dispatch` подставляет `score` по счётчику вызовов оценщика и делегирует в тот же
  диспетчер. Существующий E2E-контракт Gate 3 не сломан — расширён. **Нюанс
  переиспользования сценарной модели между прогонами** снят конструкцией
  (`ScenarioManagerModel` — свежий инстанс на диспатч `ManagerAgent`), зафиксирован в
  докстринге `MockAgentRunner` И теперь проверен вторым E2E. НЕ создан finalizer (4d).
- **4d — DONE (ждёт приёмки владельца).** Единственный handoff проекта построен
  целиком. `src/finalizer_agent.py`:
  * **Получатель — настоящий LLM-агент** `finalizer_agent` (`output_type=FinalReport`,
    `gpt-4o-mini`). Принимает управление через настоящий handoff и САМ производит
    финальное оформление (заголовок, `short_summary` аннотацией, `markdown_report`
    телом, блок follow-up, дисклеймер). **Развязка 2 — ПРИНЯТА владельцем:**
    оформление идёт ЧЕРЕЗ модель (получательство handoff = ход LLM); содержание
    восстанавливается финализатором из пришедшего через историю диалога
    `model_dump_json()` (парсинга в коде на приёмной стороне нет). Детерминированная
    граница «косметика в коде» ОТКЛОНЕНА (несовместима с механикой handoff).
    Мёртвый код `build_final_presentation` + пять косметических тестов, бивших её
    напрямую, УДАЛЕНЫ (в рантайме не вызывалась — grep подтвердил).
  * **Стык (вариант A — агент-шлюз, ПРИНЯТ владельцем)** — `finalizer_gateway`
    (`handoffs=[handoff(finalizer_agent)]`, настоящая механика handoff, не `.as_tool`)

    единственный прогон шлюза, тот делает transfer на финализатор; `ReportData`

    петля про финализатор не знает (`evaluator_loop.py` НЕ тронут), лимит её итераций

    след: петля остаётся в неведении, вся механика handoff в одном модуле-владельце).

  Четыре юнит-теста (`tests/test_finalizer.py`): схема `FinalReport` (валидность /
  обязательность поля) + стык: у шлюза настоящий `Handoff` на `FinalizerAgent`,
  handoff строго ПОСЛЕ выхода из петли и лимит итераций (`1 + MAX_ROUNDS`) не затронут.
  Решение зафиксировано в DECISIONS.md (уточнения к D-09: вариант A; природа
  финализатора — LLM; развязки 1 и 3 рассмотрены и отклонены). К UI НЕ подключён (Gate 5).
